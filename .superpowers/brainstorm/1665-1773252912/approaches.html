Asset Pipeline Architecture — 3 Approaches

How Agent 1.4 (Visual Orchestrator) hands off to Phase 2, and how Phase 2 agents coordinate

A) Sequential Pipeline

RECOMMENDED

Shot list drives everything — agents run in sequence

Agent 1.4 writes shotlist.json (narrative needs, not strict types)
    ↓
Agent 2.1 Bulk Acquisition — scrapes all sources for topic-relevant media
    ↓ writes to assets/ (unnumbered, by type folder)
Agent 2.1b Gap Analysis — maps acquired assets against shotlist, identifies gaps
    ↓ updates manifest.json with mappings + gap list
Agent 2.2 Vector Generation — generates ComfyUI prompts for needed vectors
    ↓ vectors/ folder
Agent 2.3 Remotion Animations — renders motion graphics (maps, silhouettes, diagrams)
    ↓ animations/ folder
Agent 2.4 Asset Manager — numbers everything, moves unmatched to _pool/, final manifest

Pros

  • Simple mental model — each agent has one job
  • Gap analysis naturally handles the "loose coupling" you wanted
  • Each step can be checkpointed and resumed
  • Easy to debug — if assets are wrong, you know which agent failed

Cons

  • Slower — no parallelism between acquisition and generation
  • Gap analysis is a separate step (more complexity)
  • Vector generation waits for all acquisition to finish

B) Parallel Tracks

Acquisition and generation run simultaneously from the shot list

Agent 1.4 writes shotlist.json (narrative needs)
    ↓ splits into two tracks
┌──────────────────────────────────────┐
│ Track 1: Acquisition                │ Track 2: Generation
│ 2.1 Bulk scrape by topic            │ 2.2 Vector prompts from shotlist
│ 2.1b Map against shotlist            │ 2.3 Remotion renders from shotlist
└──────────────────────────────────────┘
    ↓ both complete
Agent 2.4 Asset Manager — merges, numbers, final manifest

Pros

  • Faster — acquisition and generation happen simultaneously
  • Generation doesn't wait for scraping
  • Natural split: acquired vs generated are independent

Cons

  • No gap-aware generation — vectors/animations can't fill acquisition gaps
  • Harder to coordinate — parallel agents need shared state
  • Asset Manager has more work merging two independent outputs

C) Hybrid (Parallel + Gap-Aware)

Parallel tracks but with a gap-analysis feedback loop

Agent 1.4 writes shotlist.json (narrative needs)
    ↓ splits into two tracks
┌──────────────────────────────────────┐
│ Track 1: Acquisition                │ Track 2: Generation (known items)
│ 2.1 Bulk scrape by topic            │ 2.2/2.3 Maps, diagrams, silhouettes
│ 2.1b Gap analysis                  │ (things we KNOW we need regardless)
└──────────────────────────────────────┘
    ↓ gap analysis identifies unfilled shots
Track 2b: Gap-fill generation — additional vectors/animations for gaps
    ↓
Agent 2.4 Asset Manager — merges, numbers, final manifest

Pros

  • Fastest — parallelism where possible
  • Gap-aware — generation can fill what acquisition missed
  • Known animations (maps, silhouettes) start immediately

Cons

  • Most complex orchestration — feedback loop adds coordination
  • Three distinct phases instead of one linear flow
  • Harder to build and debug

My Recommendation: Approach A

The sequential pipeline is the right choice because:

  • Simplicity wins — this is already the most complex phase of your pipeline. Adding parallel coordination makes debugging much harder.
  • Speed isn't the bottleneck — the scraping/downloading is I/O bound, and the generation steps (ComfyUI prompts + Remotion renders) are relatively fast. The real time cost is scraping, which runs regardless.
  • Gap analysis is the key feature — the whole point of bulk-first is to know what you have before deciding what to generate. Approach A makes this natural.
  • Checkpointing is trivial — if scraping fails halfway, you resume from where you left off. No parallel state to reconcile.
  • You can upgrade later — if speed becomes a problem, A can be parallelized into C without changing the agent interfaces. The manifest.json contract stays the same.
